# AI协作开发复盘总结 Skill

## 用途

在项目开发完成后，自动生成 AI 与人类协作开发的复盘报告，总结整个开发过程中的经验、问题和改进建议。

## 触发条件

- 用户请求：`/review`、`复盘总结`、`生成复盘报告`
- 项目开发完成后
- 需要向领导/团队展示 AI 协作开发过程时

## 执行步骤

### 阶段一：信息收集

1. **扫描项目文档**
   - 检查 `.specify/` 目录下的规范文件（spec.md、plan.md、tasks.md）
   - 检查 `specs/` 目录下的设计文档
   - 检查代码变更历史（git log）

2. **分析对话记录**
   - 读取 `.claude/projects/` 下的会话记录（.jsonl 文件）
   - 提取各阶段的交互内容
   - 识别关键决策点和问题解决过程

3. **识别技术栈和工具**
   - 从 plan.md 提取技术栈
   - 从 CLAUDE.md 提取项目规则和约束
   - 从 skills 目录识别使用的辅助工具

### 阶段二：阶段划分

按照软件工程生命周期划分以下阶段：

| 阶段 | 名称 | 关键内容 |
|------|------|----------|
| 1 | 需求收集 | 用户故事、功能点、优先级定义 |
| 2 | 需求澄清 | `/speckit.clarify` 问答、边界确认 |
| 3 | 架构设计 | 技术选型、模块划分、数据模型 |
| 4 | 实施计划 | 任务分解、依赖关系、里程碑 |
| 5 | 代码实现 | 核心功能开发、单元测试 |
| 6 | 编译验证 | 构建配置、签名打包、环境问题 |
| 7 | 测试迭代 | AI自测问题（A类）、用户反馈问题（B类） |
| 8 | 交付部署 | 安装包生成、分发方式 |

### 阶段三：问题分类

#### A类：AI自我发现的问题
- AI在开发过程中主动发现并解决的问题
- 包括：代码规范、性能优化、安全漏洞、边界处理
- 格式：`A1, A2, A3...`

#### B类：用户反馈的问题
- 用户在测试/演示中发现的问题
- 包括：功能缺失、UI问题、交互体验、bug
- 格式：`B1, B2, B3...`

### 阶段五：内容填充要点

1. **量化数据**
   - 代码行数、文件数量
   - 提交次数、迭代轮次
   - 问题数量、解决时间

2. **引用规范**
   - UX 规范来源和链接
   - 设计最佳实践引用
   - 官方文档参考

3. **截图嵌入**
   - 将外部图片引用转为 base64 内嵌
   - 确保文档可离线查看

4. **格式输出**
   - 默认输出 Markdown
   - 可选导出 Word（通过 pandoc 转换）
   - 可选生成 PDF

## 报告质量要求

### 必须包含
- [ ] 完整的软件工程生命周期覆盖
- [ ] AI自我发现的问题（A类）优先于用户反馈（B类）
- [ ] 具体的问题描述和解决方案
- [ ] 量化的统计数据
- [ ] UX规范和最佳实践的引用

### 禁止内容
- [ ] 敏感信息（API Key、密码、Token）
- [ ] 无关的技术细节堆砌
- [ ] 模糊的形容词（"很好"、"不错"）
- [ ] 未经验证的推测

## 输出位置

- 默认路径：`specs/AI协作开发复盘报告_[项目类型].md`
- 命名规则：`AI协作开发复盘报告_软件工程视角.md`

## 示例调用

```
用户：帮我生成AI协作开发复盘报告
AI：开始分析项目开发过程...
    1. 收集规范文档...
    2. 分析对话记录...
    3. 生成阶段报告...
    报告已生成：specs/AI协作开发复盘报告_软件工程视角.md
```

## 注意事项

1. **隐私保护**：检查并移除所有敏感信息（Token、API Key、密码）
2. **客观记录**：如实记录问题，不隐瞒 AI 的错误
3. **时间线准确**：按照实际发生顺序记录事件
4. **可追溯性**：提供足够上下文让读者理解问题背景
